Natalie@nex311/6/2022, 8:44 AM

idk man

HTML + CSS is a lot better of a layout system than native APIs in a lot of ways. It's standard across platforms, it's nice and declarative, it's got massive amounts of accessibility features built right in. It saves a ton of dev toil not to have to write a custom native UI for every app on two separate OSes and the web, and it certainly makes it more feasible to use operating systems that aren't owned by evil megacorps. If Discord existed ten years ago it would absolutely not have had a Linux app.

It flows both ways, too. Because VSCode was implemented in Electron you can use it in a web browser for free, which is a term we programmers use to mean "a considerable amount of effort but way less than if you had to implement the whole thing from scratch". I don't even really like VSCode much as an editor but I sure use it a bunch because it lets me make some tweaks to a repo with syntax highlighting and shit in my browser, and I can even hook it up to a VM if I want to. That's rad! And it's directly possible because of Electron.

There's a lot to be said against Chromium and Blink becoming a browser monoculture, and I hope that Electron is just one step on a longer path. But the fact that you can open a dev inspector to see (and modify!) the internals of a desktop app is an unambiguously excellent thing.

I still don’t understand why it appears to take intense engineering effort to avoid making electron apps consume an order of magnitude more memory than they seem to need. What’s it actually doing.

Natalie@nex311/6/2022, 9:42 AM

I'm by no means an expert, but I'd guess the answer is several variations on the theme of "being a browser in ways that are unnecessary for a desktop app but are also difficult to disentangle from the necessary engines".

Chromium's architecture in particular is, as I understand it, heavily geared towards security by means of sharing as few resources as possible between tabs. This also means sharing as few resources as possible between tabs and the core browser process. It's easy to imagine that this ends up resulting in some amount of resource duplication between an Electron app's single tab and the underlying "browser" process, as well as additional costs to send data across the tab/core boundary.

cassie@porglezomp11/6/2022, 9:46 AM

Yeah I figured. I would love to hear from someone who is an expert because it’s an interesting problem to deal with.

Also I wish one of the companies with too much money would invest in fixing that problem for everyone.

Natalie@nex311/6/2022, 9:52 AM

The problem with "fixing it for everyone" is that Chromium is always going to be a quickly-moving target whose first priority is not desktop apps. It may be the case that the best solution is to create a new thing that is not Chromium, but then you're faced with the daunting proposition of chasing web standards instead, which are already ferociously vast and are expanding at an increasing rate—which is also a large factor in the Blink monoculture.

But given that Microsoft has hitched their ride to Electron in a number of ways, I imagine that there are a lot of people there thinking very hard about ways to make it better. That probably won't involve resurrecting Trident or whatever, but it may involve something like "dedicating a team to making Blink as decoupled from the rest of Chromium as possible" which would still probably be a good thing.
