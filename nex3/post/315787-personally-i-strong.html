Natalie@nex311/16/2022, 3:32 AM

Personally, I strongly agree that a view-sourceable, user-scriptable and -stylable web is extremely valuable. But also, part of my day job is running the CSS infrastructure team at Google. There are a number of things to understand about the realities of running very large web apps when thinking about this:

  • Bandwidth and time-to-first-render are by far the biggest concern for these transformations. Gzipping these files over HTTP definitely gives more value than minification or class renaming, but doing all of these is significantly better than just gzip.

  • A secondary consideration, but still important, is not leaking new features before they're ready. This is mostly handled by just not serving stuff we don't want users to see, but obfuscated names are still an important backstop.

  • An increasingly important consideration is being able to have multiple versions of the same CSS coexist on the same page without conflicting. Different teams may own different components that render on the same page and share some stylesheets in common, and they may need to have different release cadences.

  • In practice, the obfuscated class names in Google products (at least the ones like Gmail that don't use Angular) are as stable over time as any non-obfuscated semantic-CSS website that gets regular updates. To my mind, that makes them much more user-stylable than a website using Tailwind where the class names are guaranteed to change every time the styles change.

All this isn't to say that a user-serviceable web is impossible, but that there are real and thorny issues. Hopefully some of these can be solved by upcoming web platform improvements like CSS scoping, though!
